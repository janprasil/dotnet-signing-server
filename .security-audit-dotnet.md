# Security Audit: dotnet-signing-server

> Audited on 2026-04-04 | Scope: All source files in `/dotnet-signing-server`
> Auditor: Automated deep analysis (Claude Code)
> Severity scale: **CRITICAL** > **HIGH** > **MEDIUM** > **LOW** > **INFO**

---

## Executive Summary

The dotnet-signing-server is a multi-tenant SaaS PDF signing engine with generally good security practices: hashed API tokens with constant-time comparison, per-user tenant isolation on most endpoints, 2FA on login, PBKDF2 password hashing, input size limits, and request throttling. However, several findings require attention -- particularly around Stripe checkout replay, temporary file management, race conditions in credit debit, and missing cleanup of flow pipeline state files.

**Critical: 1 | High: 5 | Medium: 7 | Low: 5 | Info: 4**

## Findings

### CRITICAL-01: Stripe Checkout Session Confirmation -- Double-Spend / Replay

**File:** `Controllers/BillingController.cs:200-255`

The `ConfirmCheckout` endpoint adds credits based on a Stripe session ID passed as a query parameter. **There is no idempotency check** -- the same `session_id` can be replayed to add credits multiple times.

```csharp
// BillingController.cs:239 -- credits added without checking if session was already processed
user.CreditsRemaining += documents;
await _dbContext.SaveChangesAsync();
```

**Impact:** A user who completes a single payment can repeatedly hit `/Billing/Checkout/Confirm?session_id=<same_id>` to add unlimited credits.

**Mitigation:**
1. Record processed session IDs in the database (e.g., in `WebhookEvents` table) and reject duplicates.
2. Use Stripe webhooks (`checkout.session.completed`) as the authoritative credit-granting mechanism instead of relying on the redirect-based confirmation.

### HIGH-02: Race Condition in Credit Debit

**File:** `Controllers/ApiController.cs:1013-1041`

Credit checking and debiting happen in separate, non-atomic operations:

```csharp
// Check (line 1022):
if (requiredCredits > 0 && user.CreditsRemaining < requiredCredits)

// ... signing operation happens ...

// Debit (line 1038):
user.CreditsRemaining = Math.Max(0, user.CreditsRemaining - debit);
_dbContext.Users.Update(user);
await _dbContext.SaveChangesAsync();
```

**Impact:** Concurrent requests can pass the credit check simultaneously before either debit completes, allowing a user to consume more credits than they have. With 25 concurrent requests allowed per key (`LimitsOptions.MaxConcurrentRequestsPerKey`), a user with 1 credit remaining could potentially sign up to 25 documents.

**Mitigation:** Use a database-level atomic operation:
```sql
UPDATE users SET credits_remaining = GREATEST(0, credits_remaining - @debit)
WHERE id = @userId AND credits_remaining >= @debit;
```
Or use optimistic concurrency with EF Core `[ConcurrencyCheck]` / `RowVersion`.

### HIGH-03: Flow Pipeline State Files Never Cleaned Up

**File:** `Services/FlowPipelineService.cs:296-308`

Flow run state is persisted to `{AppBase}/flow-runs/{guid}.json`. These files contain:
- Full base64-encoded PDF content (potentially 20MB+ per document)
- Certificate PEM strings
- TSA credentials (username/password in plaintext)
- All flow operation data

**There is no cleanup mechanism** -- no deletion after completion, no TTL, no background purge.

```csharp
private async Task PersistAsync(FlowRunState state, CancellationToken cancellationToken)
{
    var path = Path.Combine(_root, $"{state.Id}.json");
    var json = JsonSerializer.Serialize(state, ...);
    await File.WriteAllTextAsync(path, json, cancellationToken);
}
```

**Impact:**
1. **Disk exhaustion:** Each flow run could write tens of megabytes. Over time this fills the disk.
2. **Sensitive data at rest:** PDF content, certificates, and TSA credentials persist on disk indefinitely in plaintext JSON.

**Mitigation:**
1. Delete flow state files after the flow reaches `Done` or `Error` status.
2. Implement a background cleanup task with a TTL (e.g., 1 hour for completed flows).
3. Consider storing flow state in the database instead of the filesystem.

### HIGH-04: Presigned PDF Temp Files -- No Cleanup on Abandoned Presigns

**Files:** `Services/PdfSigningService.cs:77`, `Controllers/ApiController.cs:283`

Presigned PDFs are written to the system temp directory:
```csharp
string preSignedPdfPath = IOPath.Combine(IOPath.GetTempPath(), $"presigned_{Guid.NewGuid():N}.pdf");
File.WriteAllBytes(preSignedPdfPath, pdfWithPlaceholder);
```

The file is deleted in the `/api/sign` endpoint after successful signing. However, if the client never calls `/api/sign` (abandons the flow, network error, etc.), the temp file and the `SigningData` database record persist forever. `SigningData` has no `CreatedAt` or `ExpiresAt` field.

**Impact:**
1. **Disk exhaustion** from orphaned presigned PDFs.
2. **Sensitive PDFs persist** on disk indefinitely without encryption.

**Mitigation:**
1. Add a `CreatedAt` timestamp to `SigningData`.
2. Implement a background cleanup job that purges orphaned presign records and their temp files after a TTL (e.g., 30 minutes).

### HIGH-05: TSA Credentials Stored in Database in Plaintext

**File:** `Models/SigningData.cs:13-15`

```csharp
public string? TsaUrl { get; set; }
public string? TsaUsername { get; set; }
public string? TsaPassword { get; set; }
```

When a user provides custom TSA credentials in a presign request, these are stored unencrypted in the `SigningData` database table. Combined with the lack of expiration (HIGH-04), these credentials persist indefinitely.

**Mitigation:** Encrypt TSA credentials at rest using ASP.NET Core Data Protection, or avoid storing them entirely (require them to be re-submitted in the sign request).

### MEDIUM-01: No Rate Limiting on Login / 2FA Endpoints

**File:** `Controllers/AccountController.cs:148-192, 273-313`

The login (`/Account/SignIn`) and 2FA (`/Account/TwoFactor`) endpoints have no rate limiting or account lockout. The 2FA code is a 6-digit numeric OTP valid for 10 minutes.

**Impact:** An attacker can brute-force the 2FA code (1,000,000 combinations). With no rate limit on these endpoints, this is feasible within the 10-minute window at moderate request rates. The `RequestThrottlingMiddleware` only applies to `/api` routes.

**Mitigation:**
1. Implement rate limiting per email address on login and 2FA endpoints.
2. Implement account lockout after N failed attempts.
3. Invalidate the OTP after 3-5 failed attempts.

### MEDIUM-02: `FixedTimeEquals` Short-Circuit on Length Mismatch

**File:** `Controllers/ApiController.cs:1005-1011`

```csharp
private static bool FixedTimeEquals(string left, string right)
{
    var leftBytes = Encoding.UTF8.GetBytes(left);
    var rightBytes = Encoding.UTF8.GetBytes(right);
    return leftBytes.Length == rightBytes.Length &&
           CryptographicOperations.FixedTimeEquals(leftBytes, rightBytes);
}
```

The length check `leftBytes.Length == rightBytes.Length` short-circuits before the constant-time comparison. An attacker can determine the length of the `AttachmentDebitBypassKey` by timing responses.

**Mitigation:** Hash both values with HMAC before comparing, or pad to equal lengths.

### MEDIUM-03: `ExternalSignatureContainer` Hardcodes RSA Algorithm

**File:** `Services/ExternalSignatures.cs:60`

```csharp
sgn.SetExternalSignatureValue(_signature, null, "RSA");
```

The algorithm is hardcoded to "RSA". If a client provides an ECDSA or EdDSA certificate, the signature injection will produce an invalid PKCS#7 container.

**Mitigation:** Detect the algorithm from the certificate's public key and pass the correct identifier.

### MEDIUM-05: CORS Reflects Arbitrary Request Headers

**File:** `Program.cs:229-232`

```csharp
context.Response.Headers["Access-Control-Allow-Headers"] = 
    context.Request.Headers["Access-Control-Request-Headers"].ToString()
    ?? "Authorization,Content-Type";
```

The CORS implementation reflects the requested headers back, which is equivalent to `Access-Control-Allow-Headers: *`.

**Mitigation:** Restrict to the specific headers needed: `Authorization, Content-Type, X-P4PDF-Attachment-Billing-Bypass`.

### MEDIUM-06: PFX Private Key Material Not Zeroed After Use

**File:** `Services/PdfSigningService.cs:835-861`

```csharp
private static (IX509Certificate[] Chain, ICipherParameters PrivateKey) LoadFromPfx(string pfxContent, string password)
{
    byte[] pfxBytes = Convert.FromBase64String(pfxContent); // never zeroed
    return LoadFromPfxBytes(pfxBytes, password);
}
```

PFX byte arrays containing private keys are never explicitly zeroed after use. They persist in managed memory until GC collects them, and could appear in memory dumps or swap files.

**Mitigation:** Use `CryptographicOperations.ZeroMemory()` on `pfxBytes` in a `finally` block.

### MEDIUM-07: Default TSA Uses HTTP (No TLS)

**File:** `appsettings.json:37`

```json

    "Url": "http://timestamp.sectigo.com",
```

The default TSA URL uses HTTP. Timestamp requests (containing document hashes) are sent unencrypted. Users can also provide arbitrary TSA URLs via API input, which are not validated for HTTPS.

**Mitigation:** Default to HTTPS TSA URLs. Consider warning or rejecting HTTP TSA URLs.

## Tenant Isolation Analysis

The server operates on a **per-user** model. Each API token maps to exactly one `User`. All data access is filtered by `userId`:

| Resource | Isolation check | Location |
|----------|----------------|----------|
| SigningData | `signingData.UserId != user.Id` returns 403 | `ApiController.cs:267-269` |
| Templates | `template.UserId != userId` throws | `PdfTemplateService.cs:209` |
| Flow runs | `state.UserId != userId` throws | `FlowPipelineService.cs:63` |
| API tokens | `t.UserId == userId.Value` filter | `ApiTokensController.cs:39` |
| Flow sign | `state.UserId != userId` check | `FlowPipelineService.cs:83` |

**Assessment:** User-level isolation is correctly and consistently implemented. User A cannot access User B's signing data, templates, or flows. However, there is no higher-level "organization" or "tenant" grouping.

## Presign/Sign Flow Security Analysis

```text
Client (Fortify/Browser)              Server                    TSA
       |                                |                         |
       |-- POST /api/presign ---------->|                         |
       |   {PDF, Certificate, Rect}     |                         |
       |                                |-- Write presigned.pdf -> disk
       |                                |-- Store SigningData ---> DB
       |<-- {id, hashToSign} ----------|                         |
       |                                |                         |
       |   [Client signs hash locally]  |                         |
       |                                |                         |
       |-- POST /api/sign ------------>|                         |
       |   {id, signedHash}             |                         |
       |                                |-- Verify userId match   |
       |                                |-- Read presigned.pdf <-- disk
       |                                |-- Inject signature      |
       |                                |-- Request timestamp --->|
       |                                |<-- TSA token -----------|
       |                                |-- Delete temp file      |
       |                                |-- Remove DB record      |
       |                                |-- Debit credits         |
       |<-- {signedPDF} --------------|                         |
```

**Security properties:**
- **User isolation:** `signingData.UserId != user.Id` check prevents cross-user access
- **One-time use:** `_dbContext.SigningData.Remove(signingData)` prevents replay of the same presign
- **Temp file cleanup:** Deleted after successful sign (but not on abandonment -- see HIGH-04)

**Trust boundary:** The server cannot validate that `signedHash` is a cryptographically valid signature over `hashToSign` because the private key stays on the client. This is by design for the external signing pattern. An invalid signature will result in a PDF with an invalid digital signature visible in PDF readers.

## What's Done Well

1. **Per-user data isolation** is consistently enforced across all endpoints.
2. **Token hashing** with HMAC-SHA256 and constant-time comparison is correctly implemented.
3. **Production secret enforcement** -- the app refuses to start with weak token secrets.
4. **Browser vs. server token distinction** with origin restrictions for browser tokens.
5. **IP whitelisting** support for API tokens with CIDR notation.
6. **Request throttling** and body size limits are in place.
7. **Non-root Docker container** with proper user setup.
8. **2FA via email OTP** for web portal login.
9. **Anti-forgery tokens** on all MVC form submissions.
10. **Error sanitization** -- production errors return trace IDs, not stack traces.
11. **EF Core parameterized queries** throughout -- no SQL injection risk.
12. **Template field name validation** with regex `^[A-Za-z0-9_-]+$`.
13. **Content size limits** enforced at both middleware level (body size) and application level (PDF/image/attachment).
