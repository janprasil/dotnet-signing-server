# Security Audit v2: dotnet-signing-server

> Audited on 2026-04-04 | Scope: All source files in `dotnet-signing-server/`
> Follow-up to v1 audit from same date; verifies fixes and identifies new/remaining issues
> Severity scale: **CRITICAL** > **HIGH** > **MEDIUM** > **LOW** > **INFO**

---

## Executive Summary

This is a follow-up audit after security fixes were applied to address the findings in v1. The most critical issue (Stripe checkout replay) has been fixed with an idempotency check. The credit debit race condition is now resolved with an atomic SQL UPDATE. Presigned PDF cleanup and flow state file cleanup are both implemented via a background service. TSA credentials are now encrypted at rest via DataProtection. PFX zeroing and FixedTimeEquals HMAC comparison are both fixed. CORS no longer reflects arbitrary headers.

However, several **new and remaining issues** were identified, including a missing user ownership check on checkout confirmation, the `IsActive` flag being completely ignored during auth, unauthenticated debug/test endpoints, and a concurrency gap in the idempotency check.

**Critical: 0 | High: 3 | Medium: 5 | Low: 5 | Info: 5**

## New & Remaining Findings

### HIGH-01: Checkout Confirmation Missing User Ownership Validation

**File:** `Controllers/BillingController.cs:201-275`

The `ConfirmCheckout` endpoint reads the `session_id` from the query string and credits the **currently logged-in user**. However, it never validates that the Stripe session's `metadata["userId"]` matches the current user's ID.

```csharp
// Line 234-240: Reads "documents" from session metadata
if (!session.Metadata.TryGetValue("documents", out var documentsValue) || ...)

// Line 242: Credits the current cookie user -- NOT the user from session metadata
var user = await _dbContext.Users.FirstOrDefaultAsync(u => u.Id == userId.Value);
// ...
user.CreditsRemaining += documents;
```

The session metadata contains `{ "userId": "<guid>", "documents": "100" }` (set at line 182-183), but the `userId` metadata field is never checked during confirmation.

**Attack scenario:** User A initiates checkout, gets a Stripe session. User A shares the `session_id` with User B. User B navigates to `/Billing/Checkout/Confirm?session_id=<A's session>` and gets the credits on their own account. The idempotency check prevents a second claim, but the wrong user gets the credits.

**Severity:** HIGH -- allows credit theft between users who know each other's session IDs (the session ID appears in the browser URL bar after Stripe redirect).

**Mitigation:** Add:
```csharp
if (session.Metadata.TryGetValue("userId", out var metaUserId) 
    && metaUserId != userId.Value.ToString())
{
    TempData["Error"] = "This payment session does not belong to your account.";
    return RedirectToAction(nameof(Index));
}
```

### HIGH-02: Checkout Confirmation Idempotency Check Has TOCTOU Race

**File:** `Controllers/BillingController.cs:219-265`

The idempotency check and the insert of the WebhookEvent record happen in two separate operations within the same `SaveChangesAsync()` call, but the check itself is a SELECT before the INSERT:

```csharp
// Line 219-221: Check if already processed (SELECT)
var alreadyProcessed = await _dbContext.WebhookEvents.AnyAsync(
    w => w.EventId == sessionId && w.EventType == "checkout.confirm");
if (alreadyProcessed) { ... return; }

// ... processing happens ...

// Line 250-257: Record as processed (INSERT, same SaveChangesAsync)
_dbContext.WebhookEvents.Add(new WebhookEvent { EventId = sessionId, ... });
user.CreditsRemaining += documents;
await _dbContext.SaveChangesAsync(); // Line 265
```

Two concurrent requests with the same `session_id` can both pass the `AnyAsync` check before either writes the WebhookEvent, resulting in double credit granting.

**Note:** The `WebhookEvents.EventId` has a unique index (`ApplicationDbContext.cs:100-101`), so one of the two concurrent `SaveChangesAsync()` calls will throw a unique constraint violation. However, the credit addition on line 259 is **not atomic** -- it reads the in-memory value and writes back. If the unique constraint fails, the credits from the first transaction still apply, so this is partially mitigated.

**Severity:** HIGH -- there is a narrow race window. The unique index provides a safety net, but the credit increment itself (`user.CreditsRemaining += documents`) is not atomic like the API debit is. The second request would fail with an exception caught at line 268-272, but only after the first request has already committed.

**Mitigation:**
1. Use `ExecuteSqlRawAsync` for atomic credit increment (similar to `DebitUserAsync` in `ApiController.cs:1048`).
2. Consider using a Stripe webhook (`checkout.session.completed`) as the sole credit-granting path instead of the client-side redirect.

### HIGH-03: `User.IsActive` Flag Is Never Checked

**File:** `Models/User.cs:24`, `Controllers/AccountController.cs`, `Services/ApiAuthService.cs`, `Controllers/ApiController.cs`

The `User` model has an `IsActive` boolean field (`User.cs:24`), but **no code path checks it**:
- `AccountController.SignIn` (line 186): queries by email only, no `IsActive` filter
- `AccountController.TwoFactorPost` (line 322): queries by email only
- `ApiAuthService.ValidateTokenAsync` (line 42-46): loads all non-revoked tokens, checks token hash, returns `apiToken.User` -- never checks `user.IsActive`
- `ApiController.GetAuthenticatedUserAsync` (line 973-996): queries user by ID, no `IsActive` check

**Impact:** A deactivated user can still authenticate via cookie session (if already logged in), via API tokens (which are never invalidated when a user is deactivated), and can sign new documents / access all features.

**Mitigation:** Add `IsActive` check in:
1. `AccountController.SignIn` after finding the user
2. `ApiAuthService.ValidateTokenAsync` after finding the matching token
3. `ApiController.GetAuthenticatedUserAsync` before returning the user

### MEDIUM-01: Unauthenticated Debug Endpoint in Production

**File:** `Controllers/HomeController.cs:26-38`

```csharp
[HttpGet("/debug/request")]
public IActionResult DebugRequest()
{
    return Ok(new
    {
        Scheme = Request.Scheme,
        Host = Request.Host.ToString(),
        IsHttps = Request.IsHttps,
        XForwardedProto = Request.Headers["X-Forwarded-Proto"].ToString(),
        XForwardedHost = Request.Headers["X-Forwarded-Host"].ToString(),

    });
}
```

This endpoint has no `[Authorize]` attribute, no environment check, and is registered in the `HomeController` which handles all environments. It leaks internal infrastructure details (reverse proxy configuration, forwarded headers, IP addresses).

**Mitigation:** Remove the endpoint, or gate it behind `[Authorize]` and environment check (`if (!env.IsDevelopment()) return NotFound()`).

### MEDIUM-02: Unauthenticated Loki Test Endpoint

**File:** `Program.cs:282-288`

```csharp
app.MapGet("/api/test-loki", async (LokiClient loki) =>
{
    await loki.LogAsync("info", "Loki test: info message from dotnet-signing-server");
    // ...
    return Results.Json(new { status = "ok", message = "Sent info, warn, and error to Loki" });

```

This `/api/test-loki` endpoint requires no authentication and can be called by anyone. It:
1. Confirms the logging infrastructure exists (information disclosure)
2. Can be used to flood the Loki instance with garbage log entries (denial of service on logging)
3. Is at the `/api/` path prefix, so it passes through the CORS origin check and request throttling middleware

**Mitigation:** Remove this endpoint entirely, or move it behind authentication.

### MEDIUM-04: Credit Increment on Checkout Is Not Atomic

**File:** `Controllers/BillingController.cs:259`

```csharp
user.CreditsRemaining += documents;
await _dbContext.SaveChangesAsync();
```

Unlike the debit side (which uses atomic `ExecuteSqlRawAsync` in `ApiController.cs:1048-1049`), the credit granting still uses an EF Core in-memory read-modify-write pattern. In a concurrent scenario (two tabs, or the race described in HIGH-02), this can lead to a lost update where one credit addition overwrites another.

**Mitigation:** Use atomic SQL:
```sql
UPDATE "Users" SET "CreditsRemaining" = "CreditsRemaining" + @credits WHERE "Id" = @userId
```

### MEDIUM-05: FlowPipelineService.CompleteSignaturesAsync Does Not Verify SigningData.UserId

**File:** `Services/FlowPipelineService.cs:105-109`

```csharp
var signingData = await _dbContext.SigningData.FindAsync(pending.SigningDataId);
if (signingData == null)
{
    throw new InvalidOperationException($"Signing data not found: {pending.SigningDataId}");
}
// No check: signingData.UserId != userId
```

While the flow state file itself is validated against `userId` (line 86), the individual `SigningData` records fetched from the database are not checked for ownership. The signing data IDs come from the flow state file (which is trusted after the user check), so this is a defense-in-depth concern rather than an exploitable vulnerability. However, if the flow state file were somehow corrupted or tampered with, this could allow reading another user's presigned PDF from disk.

**Current risk:** Low, because the flow state file provides the `pending.SigningDataId` values, and the file is only written by the server.

**Mitigation:** Add `if (signingData.UserId != userId) throw` for defense in depth.

### LOW-03: No Validation on User-Provided TSA URLs

**Files:** `Models/PreSignInput.cs`, `Models/DocumentTimestampInput.cs`, `Services/PdfSigningService.cs`

Users can provide arbitrary TSA URLs via API input fields (`TsaUrl`). While the default TSA is now HTTPS (fix from v1), user-supplied URLs are not validated:
- No HTTPS enforcement
- No URL scheme validation (could be `file://`, `ftp://`, etc.)
- Potential SSRF vector if the signing server makes outbound requests to attacker-controlled URLs

**Mitigation:** Validate that user-provided TSA URLs use HTTPS scheme only. Consider maintaining an allowlist of known TSA providers.

### LOW-04: In-Memory Rate Limiter Lost on Restart

**File:** `Controllers/AccountController.cs:23-25`

```csharp
private static readonly ConcurrentDictionary<string, (int Count, DateTime WindowStart)> _loginAttempts = new();
```

The rate limiter uses a static in-memory dictionary. State is lost on application restart, and is not shared across multiple instances in a scaled deployment. An attacker could time attacks to coincide with deployments.

**Mitigation:** This is acceptable for single-instance deployments. For multi-instance, consider Redis-backed rate limiting. Also consider adding periodic cleanup of expired entries to prevent memory growth.

### LOW-05: Flow State Files Contain Sensitive Data in Plaintext

**File:** `Services/FlowPipelineService.cs:349-354`

Although flow state files are now cleaned up (fix from v1), during their lifetime (up to 2 hours for completed flows, or until completion) they contain:
- Full base64-encoded PDF content
- Certificate PEM strings
- TSA credentials (which are now DataProtection-encrypted in the DB, but the flow state file serializes the `OriginalInput` which contains plaintext TSA creds from the original request)

```csharp
// Line 48-55: OriginalInput = input  -- this contains TsaUrl, TsaUsername, TsaPassword from the request
var state = new FlowRunState
{
    OriginalInput = input,  // contains plaintext TSA credentials
    // ...
};
await PersistAsync(state, cancellationToken);
```

**Mitigation:** Strip or encrypt sensitive fields before persisting the flow state to disk.

### INFO-03: Authentication Architecture

```text
Web Portal (Cookie Auth)                    API (Bearer Token Auth)
         |                                           |
    AccountController                          ApiController
    - Email/password login                     - GetAuthenticatedUserAsync()
    - 2FA via email OTP                          - Tries cookie auth first
    - Rate limited (5/min)                       - Falls back to API token
    - Cookie: SameSite=Lax, Secure=Always        - ApiAuthService.ValidateTokenAsync()
         |                                           |
    [Missing: IsActive check]                  [Missing: IsActive check]
```

Token validation flow (`ApiAuthService.cs:24-78`):
1. Extract Bearer token from header
2. Load ALL non-revoked, non-expired tokens from DB (line 42-44) -- **performance note: this loads all tokens into memory**
3. Compare token hash against each candidate using `FixedTimeEquals` (line 47)
4. For browser tokens: validate Origin against allowed origins
5. For server tokens: reject if non-local Origin is present
6. Check IP whitelist if configured
7. Return the associated User

**Performance concern:** Step 2 loads all active API tokens into memory on every request. As the user base grows, this becomes an O(N) scan. Consider caching or indexing.

## What's Done Well (Updated)

1. **Per-user data isolation** is consistently enforced across signing, templates, flows, tokens, and billing.
2. **Atomic credit debit** with SQL `WHERE CreditsRemaining >= @debit` prevents going negative.
3. **Token hashing** with HMAC-SHA256 and `FixedTimeEquals` is correctly implemented.
4. **Production secret enforcement** -- app refuses to start with weak token secrets.
5. **Browser vs. server token distinction** with origin restrictions for browser tokens.
6. **IP whitelisting** with CIDR support for API tokens.
7. **Request throttling** (25 concurrent per user/IP) and body size limits (40MB).
8. **Non-root Docker container**.
9. **2FA via email OTP** with rate limiting on login and 2FA endpoints.
10. **Anti-forgery tokens** on all MVC form submissions.
11. **Error sanitization** -- production errors return trace IDs, not stack traces.
12. **EF Core parameterized queries** throughout -- no SQL injection risk.
13. **DataProtection encryption** of TSA credentials at rest.
14. **PFX byte zeroing** after use with `CryptographicOperations.ZeroMemory`.
15. **Background cleanup** of orphaned presigned PDFs and flow state files.
16. **Content size limits** at both middleware and application level.
17. **Certificate algorithm auto-detection** (RSA/ECDSA) in external signature flow.
18. **Hardcoded CORS allowed headers** (no reflection).
19. **Unique index on WebhookEvents.EventId** as safety net for idempotency.
